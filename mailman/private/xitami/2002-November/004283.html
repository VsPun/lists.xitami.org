I was only asking this because I do not personally own XiPro, but know
about the implications of iMatix not spending 2-3 hours to re-compile XiPro
and distribute the fixed version.

Exploit code does exist for the OpenSSL/SSLeay security holes, but I am not
sure how widely available it is.  The fact that such code exists basically
blows any security measures completely out of the water that you (and
others) have taken to create a secure environment.  If done properly,
exploit code can allow OpenSSL to run as usual and snag plain-text
transfers and forward them to some other location...e.g. unencrypted free
credit card numbers, addresses, e-mail, personal information, etc.  It
would take weeks/months/years(?) to figure out what happened since the
attacker is a remote machine acting passively except for one request.
Basically, this allows for something the equivalent of CodeRed I and II for
Xitami, Apache, and anything else not running on a fixed OpenSSL code base.

The best solution is probably to complain to iMatix that you paid good
money for a good product and you want at least some sort of fix for this
problem (even if they just recompile and statically link OpenSSL v0.9.6g
with XiPro, it would be better than doing nothing at all).
