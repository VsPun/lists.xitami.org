hhhmmm...heres a thought.
 
Currently I have a filter processing .pl files with perl anywhere in my webpages (htdocs) directory.  Executing them just as they would be executed in the cgi-bin directory.  Create another filter for "*.ptml" files to be executed by perl.  Creating the static urls from dynamic content is easy...however its the conversion back during request processing that has to happen within the script that was executed.
 
Dynamic URL: /cgi-bin/foo.pl?var1=10&var2=30&var3=30
 
would be translated to the static URL in a page, this can be easily done in the script that is would have normallhy generated the dynamic URL:
 
/cgi/var1_10/var2_20/var3_30/foo.phtml
or 
/cgi/10/20/30/foo.phtml
 
A filter/alias would have to execute the *.phtml files from a base directory...cause the above static urls would break with a 404 because the actual path /cgi/10/20/30 (or whatever) doesn't actually exist.  Then foo.phtml, processed as a perl script, would take the $QUERY_STRING and parse it on the "/" thus returning the first few variables.
 
I think the key here is getting xitami to see the *.ptml file extension first and look for that file in a specific directory and execute it before xitami tried to look for the file in the full path/url that doesn't actually exist.  This still might actually work using the cgi-bin directory....b/c Xitami has an option for the cgi files - execute with /cgi-bin as the root of the path or execute when /cgi-bin is anywhere in the path...though it typically still looks for an exact path doesn't it?
