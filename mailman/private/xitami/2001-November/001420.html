Frances Turner wrote:

>I think what you have is an FTP client problem related to Network
>Address Translation (NAT). The problem is this:

>Netscape's FTP client issues a command (SYST I think) to the FTP server
>to tell it what IP address it has and then tries to connect to that IP
>address for future connections. This is stupid because the FTP cleint
>already has an IP address that works. Of course this IP address is
>embedded within the message so the NAT s/w on the router doesn't modify it.

I ran my copy of LeechFTP, a simple windows-based client to test Frances's
theory. Indeed, I did see that the software issued a SYST command. Of
course, on my side of the router, it doesn't cause any problems. It is only
a problem for people on the outside of my network.

BTW, the possibility that the FTP client was causing the server to send my
internal IP is the original reason why I wanted to bag the TZO "IP catching"
client software and use xitami's imbedded client to retreive the IP. My
thought was that Xitami would pass the correct IP to the FTP server and
would send to the requesting FTP client my true "internet" IP, instead of
the IP I use on this side of the router.

One user reported that he could upload to me, but not download from me. This
is the person that reported seeing my local IP address, instead of my global
address as he should have seen it. Frances, is this consistent with your
diagnosis? Do you think there is a solution? Anyone else have any ideas?

Thanks.
