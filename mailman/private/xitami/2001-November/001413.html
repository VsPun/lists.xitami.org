I think what you have is an FTP client problem related to Network 
Address Translation (NAT). It is not a DDNS problem a Xitami problem as 
it will occur with other FTP servers.

Try getting a remote machine to run a very simple client (e.g. the CLI 
ftp client included with windows 9x). It should have no problem, whereas 
the funky GUI client may fail. I have seen this behavious with Netscape 
4.x and believe it also occurs with other clients. The problem is this:

Netscape's FTP client issues a command (SYST I think) to the FTP server 
to tell it what IP address it has and then tries to connect to that IP 
address for future connections. This is stupid because the FTP cleint 
already has an IP address that works. Of course this IP address is 
embedded within the message so the NAT s/w on the router doesn't modify it.

The way to check this is (assuming the simple DOS ftp worked) is to look 
and see if the GUI client does a SYST command or not and what the 
response is.
