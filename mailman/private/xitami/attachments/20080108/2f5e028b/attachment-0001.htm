Currently I have a filter processing .pl files with perl anywhere in my webpages (htdocs) directory.&nbsp; Executing them just as they would be executed in the cgi-bin directory.&nbsp; Create another filter for "*.ptml" files to be executed by perl.&nbsp; Creating the static urls from dynamic content is easy...however its the conversion back during request processing that has to happen within the script that was executed.<BR>
&nbsp;<BR>
Dynamic URL: /cgi-bin/foo.pl?var1=10&amp;var2=30&amp;var3=30<BR>
&nbsp;<BR>
would be translated to the static URL in a page, this can be easily done in the script that is would have normallhy generated the dynamic URL:<BR>
&nbsp;<BR>
/cgi/var1_10/var2_20/var3_30/foo.phtml<BR>
or <BR>
/cgi/10/20/30/foo.phtml<BR>
&nbsp;<BR>
A filter/alias would have to execute the&nbsp;*.phtml files from a base directory...cause the above static urls would break with a 404 because the actual path /cgi/10/20/30 (or whatever) doesn't actually exist.&nbsp; Then foo.phtml, processed as a perl script, would take the $QUERY_STRING and parse it on the "/" thus returning the first few variables.<BR>
&nbsp;<BR>
I think the key here is getting xitami to see the *.ptml file extension first and look for that file in a specific directory and execute it before xitami tried to look for the file in&nbsp;the full path/url that doesn't actually exist.&nbsp; This still might actually work using the cgi-bin directory....b/c Xitami has an option for the cgi files - execute with /cgi-bin as the root of the path or execute when /cgi-bin is anywhere in the path...though it typically still looks for an exact path doesn't it?<BR>
