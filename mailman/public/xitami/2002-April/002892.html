Sean,

Maybe this quote from the FAQ helps...:

"
15: I get 'Not authorized to access this resource' on other URLs =20

There are a number of specific reasons why Xitami may return this:=20

  1.. The resource you are accessing is protected by a 'webmask' =
definition which excludes your PC.=20
  2.. You are trying to browse a directory, but directory listings have =
not been enabled.=20
  3.. You are trying to use an HTTP update method (PUT, DELETE, MOVE, or =
COPY) on a resource which has not been authorised for this.=20
  4.. You are trying to run the WBA, but this has been disabled by the =
security:admin option.=20
  5.. The URL refers to a Windows short filename when a long filename =
also exists. Xitami rejects this as being a potential security =
violation. You can switch this behaviour off by setting =
security:safepaths to 0.=20
  6.. You're trying to do a HTTP PUT, DELETE, MOVE, or COPY operation, =
and specified an absolute Content-Location: or Destination: URI which =
did not have the same path as the request URI. Xitami rejects this =
because the authorisation was carried-out on the request URL, and using =
a different path for the Content-Location: or Destination: URL would =
violate this.=20
  7.. A HTTP PUT, DELETE, MOVE, or COPY operation failed because of an =
access violation at the operating system level - for instance Xitami =
tried to write to a file which was marked as read-only.=20
In most of these cases, the reason for the error will be logged when you =
enable server debugging.=20

"

Kind Regards,
Paul Claessen
