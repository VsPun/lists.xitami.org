At 02:33 PM 11/22/2004 -0500, Ron Ackerman writeth:
>Good info, now how do I fix it?

It really depends on your setup.  It sounds like you might have a really
complex setup, so it might help to try something simple first like a simple
PHP page that does nothing but spit back an '<? echo "hi\n"; ?>' to the
client.  The next thing I would do is take the server off the Internet for
a minute or two to try the same tests locally to rule out a bunch of people
just requesting pages.  If you have a really high-volume site that can't go
down, that's probably your problem and you can't do much about it unless
you want to learn C/C++ and edit the source code.  I'm pretty sure Xitami
will simply overwrite the FLT files when it loops back around to
pipe0000.flt.  If you want, you can always set up some script that
regularly goes through the directory and cleans out the FLT files.
NT-based OSes will stop the process when you hit a file in use.

>It is leaving every page visited in the pipexxxx.flt temp file on the
>server. Is this a problem with Xitami or PHP?

This is likely a Xitami issue except for if you have long-running PHP
processes (30 seconds or more).  If PHP did this, you would hear everyone
hollering about it on Slashdot.

One question that just popped into my head:  Did you accidentally turn on
the "log CGI page requests" option in the configuration?  I seem to recall
that the option keeps around the FLT files in that case and seems to be
what you are describing.
