You don't need three copies of Windows.  You need 1 copy of NT Server
or 2000 (assuming that you _have_ to use Windows due to the software you
need; any modern Unix/Linux system can do this just as easily).  Then you
bind three IP addresses to the _same_ server (NT Server and 2000 can do
this; 95/98/ME, etc cannot).  Then you configure Xitami, and whatever
other programs that you want to listen on _one_ of those IP addresses
(each).  (Xitami can be configured to do this with the [server] ipaddress
setting; by default it is "*" which means "everything you can find".)

If you have only one IP address to play with (and it sounds like you
don't, if setting three machines up is a feasible option), then you need
to look at doing some sort of selective request forwarding -- ie, having
a program that handles the request, connects to the "real thing", and
then passes back the reply it gets.  With Xitami something like LRWP
would be required for this, or if you're lucky with the URLs used a
filter or CGI script.  (Filters are basically CGI, but run with the
input being the original page before filtering -- you can ignore that
and make it all up yourself if you want.)

If you don't care about all your users that are behind corporate
firewalls being blocked from accessing it, you can run your other
servers on off-numbered ports, and send redirects.  If you need to do
this only for a few basic URLs a CGI or filter would be sufficient for
this; just have it return a Location: header with the new location.
(But any user behind a corporate firewall that says port-80-outside
only, will get redirected, but otherwise not be able to access the
service.)

As I commented in my earlier message this is not a solution to the
codered type problem, as you'll just end up causing more work for your
system not less.  Only by having another machine in front (eg, a
transparent proxy or a reverse proxy) that can do that filtering off do
you save anything.
