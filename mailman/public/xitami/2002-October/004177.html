As I was the original poster of the script example which included the
"3D"'s, I am very interested in this explanation. At the time, I thought the
spurious characters had something to do with me posting in HTML. Long after
the original incident, I discovered that all the example files from the
original source were riddled with "solid black rectangle characters" (when
viewed in Notepad) which had to be removed before the examples were usable
in a Win 2000 environment.
I suspect the mysterious appearance of the "3D"'s in my post and the
rectangles in the original files are related somehow. The files came
originally from a Perl book author's web site, and (I believe) were designed
to be run on a Unix-based web server. Subsequent to all this, I have
discovered that many other Perl authors provide separate script file
downloads for Windows vs. Unix. When I use example scripts designated for
Windows, I have no problems.
A colleague and I have discovered that if the original files are run through
a utility called Unix2DOS.com, then reversed through DOS2Unix.com, then
reversed through Unix2DOS.com *again*, the offending characters are finally
eradicated and the files are usable. I have also discovered that if these
original files are opened in WordPad, then saved with File->SaveAs to force
an overwrite, the resulting file is "cleansed" and usable on a Windows-based
web server.
I am relating all this in hopes of getting some practical insight on how the
files' originator (the book author) could have avoided all this in the first
place, as she appears to be as mystified by all this as I am. Should this
author be offering two sets of files, one for Windows-based web servers, and
the other for Unix-based web servers? Is there a better way?

> The above stems from someone having used "quoted-printable" format when
> making a posting, and for the archiving software being ignorant of this
> format, and including the content verbatim.
>
> Quoted-printable is sometimes used by some mailers when a character is
> required in the text that is not "seven-bit clean" (the relevant
> standards for e-mail on the Internet explicitly state that the content of
> messages shall consist ONLY of the characters in the SEVEN-bit ASCII
> character set, although extensions exist to handle, e.g., characters with
> the eighth bit set high).  One of these extension mechanisms is the very
> crude quoted-printable format; use of this format is generally
> deprecated.
>
> So such characters are sent by their equivalent hexadecimal code for the
> two nibbles in the octet, preceded by an equals sign.  Thus the code for
> the British "pound sign" might appear as =A3, since this character has
> the code 0xA3 (or 163 in decimal)[1].  Since the equals character is
> hijacked under this mechanism to introduce such triglyph codes for a
> single character, then occurrences of the equals sign itself will appear
> encoded, as =3D, since 0x3d or decimal 61 is the ASCII code for the
> equals sign.  (Messages in quoted-printable will sometimes also have =20
> appearing at the ends of lines, to indicate that there is actually a
> space character present there before the end-of-line.)
>
> You'll note that this mail does NOT have the requisite MIME header to
> indicate that quoted-printable characters appear herein, and therefore
> everyone's mail-readers /ought/ to see the actual triglyph sequences
> themselves, rather than the character which each sequence represents.
>
> [1] Well, that's the code in a widely-used codepage associated with
> Windows Latin-1 alphabet.  For those of us that use CP437 or CP850, we
> see a lowercase-letter-u-with-acute-accent, but that's just another of
> the vagaries of Windows software assuming that everyone is using it, and
> not including the other requisite headers to permit of automatic
> translation at the recipent's computer.
