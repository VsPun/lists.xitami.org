AFAIK, Xitami passes the .php-file which is to "execute" directly to the 
php.exe as a command line parameter and simply returns the output.
This problem is known for some time, though (several month), and there 
has not been any real workaround. If you are forced to use aliases bad luck.
I solved the problem by using virtual name-based hosts instead of 
aliased directories if the files are i.e. located on a file server. This 
works fine (as long as you won't use aliases below the virtual hosts 
level, of course ;-) ).
So the problem seems to be in the alias codepart of Xitami so that it 
won't handle any files which are to be passed to a program (i.e. filter 
like php.exe) correctly. I didn't try if perlssi or any other filter 
works because php is the only filter I'm using.
