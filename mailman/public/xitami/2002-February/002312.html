Hmm, I've never used a mailer that has the capability of deleting traffic
on a "thread" that has not yet arrived.  How do you ensure that mail is
threaded anyway, given that the use of In-Reply-To is only sketchily
followed by some MUAs, and that others use References instead, which is
something that actually belongs to Usenet?  (RFC-2822, which is still
only a "proposed standard", mentions References for the first time; it
was not part of THE standard RFC-822.  Moreover, the latter never
mandated the use of I-R-T, and even the new one only places both at the
level of a SHOULD, and not a MUST.)

Perhaps the amount of traffic on this list could justify a proper Usenet
newsgroup.  Perhaps a case could be made for restructuring the
comp.infosystems.www.servers.* hierarchy, which (foolishly, IMHO)
contains the groups:
    comp.infosystems.www.servers.mac
    comp.infosystems.www.servers.misc
    comp.infosystems.www.servers.ms-windows
    comp.infosystems.www.servers.unix

Obviously no one envisaged the concept of a cross-platform server; all
the same comp.infosystems.www.servers.xitami might not be beyond the
realms of possibility.

Would anyone care to try raising an RFD?

(OTOH, Imatix could run an NNTP server on their systems, that handles
only a xitami newsgroup, or hierarchy of them.  I'm generally against the
concept of special-to-one-server-only newsgroups, since I consider that
the advantage of a newsgroup is that one reads it from the most local
server, wherever one happens to be.  But Borland use one, as do Serenity
Systems (for the ecomstation.* hierarchy).)
