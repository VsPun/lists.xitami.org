In message <000701c0f953$7b0faac0$830a5f82@hm.uwa.edu.au>, "Don Gordon" writes:
>The HTML file is located in a password protected directory while the sound
>file (109k) and movie file (188k) are in subdirectories of the above. The
>problem does not occur in Netscape and is occasionally fixed if the page is
>refreshed in IE, but not always. I have also experimented using .ra as well
>as .rpm files, no difference. 

Changing file type probably won't help; and if you change to the real
"this is where to get the audio/video" index files (.rpu? I forget),
then the problem wil probably get worse.

>The problem does not occur if the web page has only one RealAudio file or 
>if tested on the same computer as the server. I have also tried setting 
>same usernames and passwords to the subdirectories with no effect.

The default for subdirectories is to use the settings of the parent that
is protected, and this seems to be what is happening in your case.
However I'm puzzled by your statement that it doesn't happen if there is
only one RealAudio file on the page; the factors I gave before would
leave it either working or not working with 1 or more files on the page.

What you might want to try doing is turning on the Xitami server
debugging ([Server] debug=1), and then try loading the page.  Once
you've done that look at debug/debug.log to see what queries Xitami
received, and how it handled them.  My guess is that for some reason one
of the queries must be missing the username/password -- but I'm puzzled
why including two real files on the page causes that when including just
one doesn't.
