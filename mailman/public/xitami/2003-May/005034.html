> why Xitami corrupts an EXE file on Download. HTTP.

Never does that for me. I stash huge self-checking EXE files
(Netscape 4.7, NT4 SP6a) on my server for local convenience. NT4SP6a
is over 30Megs and does a thorough self-check when run.

Xitami on various P133-P350 boxes, mostly NT4, mostly with 10-speed
net-cards to 100-speed switches to a 10-speed switch to a 10-speed
router, optic over to the library and then off into some Gigabyte
Cloud.

Yes, I have FTP enabled, turned-on, and use it. The FTP module is
weak, and a big FTP job will sometimes lock-up Xitami. But HTTP
hardly-ever burps as long as nobody has done a big FTP job today,
and I can't remember ever getting a corrupt HTTP download.

As others have suggested: there are cards with timing bugs, which
may not appear if there is a slower bottleneck along the path but
can corrupt packets if they try to run full-speed.
