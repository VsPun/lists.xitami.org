Two things come to mind:
 
1: If the above is true, then the owner of the fileserver can easily write a script that parses the directories based on subject and then on date and time.  This will chronologically order all threads based on subject.  Process each file in order as a text file...grab the author, date, time, subject and content (new content...not older repeated content) and then store that info into a DB with assigned record IDs.  The first article of a grouping based on subject will the master recordID, all subsequent same subjected articles will use the maser recordID to associate with as a common thread.  When a new subject is processed then that means a new recordID must be assigned and repeat the process.  This wouldn't necessarily be the cleanest method but I am certain it would get about 90% of everything.
2: Someone writes a LWRP script to connect to the webserver, grab all the links by "subject" from the index listing (going all the back to May 2001) and then connects to every HTML page strip out the author, subject, date, etc and store it a DB....using some of the same principles and methods from above.
Personally I think #1 would be easier assuming that directory and file structure is static.  Let me know and I will try to cook up a quick perl script that does some of the basic stuff.  All you would need to do is create the tables and db structure based on the notes provided in the script.  I would need to see a directory structure listing and the format of the file each thread is stored in.
