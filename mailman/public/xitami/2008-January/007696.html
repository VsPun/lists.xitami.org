I discovered you can also set your default page to "/"

And then create a cgi alias for
     "/" = /xitami/cgi-bin/somescript.pl
You can then use the secondary and teriary default fields for other static 
pages such as "index.html" or "defualt.html"

This will allow your landing page to a be a cgi/perl script in a central 
directory without having to create a filter for script extensions to work 
out side of your cgi directory.  Previously I had a "foo.pl" perl script in 
my root htdocs directory (/xitami/htdocs) and then had "foo.pl" as my 
default page - I needed a filter binding *.pl = /perl/bin/perl to execute my 
landing page....this posed a few maintenance and security problems.

However, the about "/" method also seems to hide directory listings of URL 
directories that do not have a default page.  Though the files in URL 
directories are still individually accessable, the directory is no longer 
viewable as a file/folder listing. This is a great way to protect 
directories from being viewed as file listings if a default page doesn't 
exist in that folder.

I have not found any downsides to this method...anyone else used/seen it 
before to comment on it?

With out any default page in the URL directories:
IE: www.foo.com/images/bar/helloworld.jpg - would still show the image in 
the browser
but : www.foo.com/images or www.foo.com/images/bar  would only yeild the 
following results

File list of /images///////////////////////
directory
Name                Size         Last 
Modified--------------------------------------------------------------------------------/images/////////////////////////
and

File list of /images/bar//////////////////////
directory
Name                Size         Last 
Modified--------------------------------------------------------------------------------/images/bar////////////////////////
